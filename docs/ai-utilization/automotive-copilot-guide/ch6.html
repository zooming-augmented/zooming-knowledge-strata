第6章 深堀

GHEC + Copilot の開発活動・ALM統合・構成比較

この章のポイント：
  • ALM（Codebeamer等）＋ GHEC の前提構成で Copilot が担える開発活動を網羅整理
  • BMW・Woven by Toyota・Mercedes-Benz など OEM事例の実態と公開情報の限界 を整理
  • GHEC + Copilot を軸に 11構成の比較マトリクスと星取表
  • 選定フローチャートで 自組織に合う構成を判断
前提アーキテクチャ（本章の参照軸）：
本章は以下の3層構成を想定している。詳細はガイド第4章を参照。

[PLM] システム要求・BOM・CAD/メカ/エレキ設計
　　↕ OSLC / REST API
[ALM: Codebeamer等] SW要求 → SWアーキテクチャ → 統合テスト・ASPICE証跡
　　↕ REST API / Webhook / commit message連携
[VCS: GHEC] SW部品単位：設計書・コード・単体テスト・PR・CI/CD
　　↑ ここにGitHub Copilot等のAIを活用

Part 1｜GHEC + Copilot で実行・効率化できる開発活動

1.1 設計書作成・管理

活動機能成熟度制約・注意点
Markdown設計書の自動生成Copilot Chat / Agent でMermaid図・技術ドキュメント生成Preview〜GA生成内容の正確性は人手確認必須。MISRA準拠の保証はない
仕様変更時の設計書更新提案Agentモードで「変更があれば設計書も更新」と指示Preview参考提案。最終確認は設計者が実施
設計書とコードの整合性チェックCIツールや手動レビューと組み合わせ—Copilot単体の自動チェックは限定的

1.2 コーディング支援

活動機能成熟度制約・注意点
インライン補完（C/C++）VS Code / JetBrains等のIDE拡張GAMISRA/社内規約準拠は静的解析ツールで別途チェック要
Copilot Chat（コード説明・提案）IDE内またはGitHub WebGA正確性は文脈に依存。人手確認必須
Coding Agent（自律タスク処理）リポジトリ解析・実装〜テスト〜エラー修正GAプロンプト設計が重要。安全性確保にレビュー必須
リポジトリ全体コンテキスト活用Copilot Spacesで設計書・コードを横断参照GA64Kトークンウィンドウ制約。バイナリ非対応

1.3 コードレビュー・品質チェック

活動機能成熟度制約・注意点
Copilot PR ReviewPRに自動コメント・改善提案。適用可能な修正案をワンクリック反映Preview〜GAカスタム指示設定で規約スタイルの強制可能
MISRA/AUTOSAR規約チェックGitHub CodeQLのAUTOSAR C++/MISRA C/C++クエリGACodeQLはISO 26262ツール認定取得済。クエリ外ルールはカバーされない
セキュリティ脆弱性検出GHAS: Code Scanning / Secret Scanning / DependabotGAプライベートリポジトリは有償ライセンス必要
静的解析ツール（PRQA/Polyspace等）連携GitHub Actions経由でCIに組み込み・結果をGitHubに反映設定要Copilot直接連携はなし。CI経由で結果を参照

1.4 テスト支援

活動機能成熟度制約・注意点
単体テスト自動生成Copilot Chatで関数・クラスのテストコード生成GA生成テストの期待値検証は必須
テストカバレッジ分析GitHub Actions + Codecov等との組み合わせ設定要Copilotの自動提案機能はなし。カバレッジレポートを参照して追加
CI失敗時のAI解析CIログをCopilot Chatに投入し原因推論GA推測含む。最終判断は開発者

1.5 トレーサビリティ管理（ALM-VCS間）

活動機能成熟度制約・注意点
コミットメッセージ規約チェックGitHub ActionsでRefs: #REQ-XXX形式を強制チェックGAルール遵守は自動検査が必要。Copilotは補助的にメッセージ提案
PRテンプレートによる要求リンク.github/PULL_REQUEST_TEMPLATE.mdにALM要求ID入力欄を設置GAGitHub側の強制機能はなし。運用ルールとの組み合わせ
Codebeamer ↔ GHEC リンク管理REST API + Webhook: commit/PR/branchをCodebeamerに同期GACodebeamer X #1234形式をcommit messageに埋め込む
CopilotによるトレースリンクAI提案Copilot Chatに「このコードが関連する要件は？」と質問研究的利用誤提案リスクあり。参考情報として扱う

Codebeamer ↔ GHEC 連携の実装パターン（公開ドキュメントより）：

  1. 要求・変更・テスト・監査証跡 = Codebeamer（正本）
  2. コード・PR・CI = GHEC（正本）
  3. commit messageに要求IDを埋め込み（Codebeamer X #要求ID Comment）
  4. Webhook / REST APIでCodebeamerへ同期
  5. CodebeamerからGitHub側実装痕跡を辿るトレーサビリティが成立
  6. OSLC = CodebeamerとPLM（Windchill等）の接続に使用（GitHubとの接続には不使用）

1.6 知識管理・ナレッジ再利用

活動機能成熟度制約・注意点
Copilot Spaces（旧Knowledge Base）複数リポジトリ横断でMarkdown設計書・仕様書等を集約しCopilot参照GAKB機能は2025年8月廃止→Copilot Spacesへ移行
Issue/PR/Discussion活用設計意図・過去議論の検索・再利用GAGitHub検索 + Copilot Chat（リポジトリ探索）で参照可能
過去バグ・修正パターン再利用Issue/commit履歴からパターン参照設定要プロジェクト固有の知識共有ルール設計が重要

1.7 ドキュメント自動生成

活動機能成熟度制約・注意点
API/I/F仕様書生成Copilot Agentが仕様データからYAML等を生成Previewテンプレート設計と生成結果の手動確認が必要
リリースノート自動生成GitHub自動リリースノート機能 / Conventional Commits連動GAタグ付けとの連携でCHANGELOG.md自動更新可能
コードコメント・API仕様生成Copilot Chatでコードコメント・docstring生成GA安全クリティカルなコードへの適用は人手確認必須

1.8 セキュリティ・コンプライアンス

活動機能成熟度制約・注意点
GitHub Advanced Security (GHAS)CodeQL（脆弱性）/ Secret Scanning / DependabotGAプライベートリポジトリは有償
SBOM生成「Export SBOM」でNTIA準拠SPDX形式をワンクリック出力GA全クラウドリポジトリで無償提供
ライセンスコンプライアンス依存関係グラフ + REUSE等のCIチェック設定要Copilot直接機能はなし
ISO 26262 / ASPICE対応CodeQL ISO 26262ツール認定取得済GA認定ツールとしての使用は条件あり

1.9 CI/CDパイプライン

活動機能成熟度制約・注意点
CI/CD自動化GitHub Actionsでビルド・テスト・静的解析・デプロイGACodebeamer Actions連携でALMワークフローからトリガー可能
AIによるパイプライン提案Copilot CLI / Agentに「Actionsワークフロー作成」を依頼Preview生成結果は手動修正・セキュリティチェック必須
リリース承認ワークフローGitHub Environments + CODEOWNERS + 手動承認ステップGAALM側承認と連動するWebhook設計が必要

1.10 AI統治の原則

どの構成でも以下を遵守すること（詳細は第5章参照）：

AIが担う領域（自動化可）人間が保持する責務（AI代替不可）
関連ドキュメント検索・要約 / 設計・コードレビュー支援 / テストケースシナリオ抽出 / 変更影響分析参照 / マルチシステム横断参照 正式な要求承認 / 安全クリティカルな判断 / 公式テスト合否判定 / 監査証跡の確定 / 契約・顧客承認

3.2 構成A（推奨）の詳細：GHEC + Copilot Enterprise

強み：

  • GHECがTISAX AL2認定取得済み（2024年10月）→ 自動車業界のセキュリティ要件をクリア （GitHub Changelog）
  • Copilot Business/EnterpriseがISO 27001認証スコープに含まれる
  • Mercedes-Benz・Woven by Toyota等OEM採用実績が最も豊富
  • Copilot Spacesによるリポジトリ横断ナレッジ活用（GA）
  • CodeQL（MISRA/AUTOSAR C++クエリ）ISO 26262ツール認定取得済み
  • SBOM生成（SPDX形式）が無償提供
  • Codebeamer ↔ GHEC連携：REST API + Webhookで実績ある接続パターンが存在

弱み：

  • AI推論はSaaS依存（モデルを社内に閉じられない）
  • Coding AgentはPreviewが多く、安全クリティカルな用途は人手確認が必須
  • C/C++のMISRA完全自動準拠は保証されない（静的解析ツールとの組み合わせ必須）

車載開発での適合性： ★★★★★（最も実績のある構成）

3.3 構成B：GHEC + Copilot + Claude Code

AにClaude Code（Anthropic製CLI/IDEエージェント）を追加する構成。

  • Claude Codeの強み：コードベース横断推論、ターミナル中心のagentic作業、PR/CI連携
  • ユースケース：大規模リファクタリング、複数リポジトリ横断影響分析、設計書とコードの整合チェック
  • 費用：Claude APIの従量課金が追加（$20+/月相当）
  • 注意：CLI中心のため、開発者ワークフローへの統合方法の設計が必要

車載開発での適合性： ★★★★☆（強力だが運用設計コストが追加）

3.4 構成E（競合）：GitLab Ultimate + Duo Enterprise

GitHub Copilotとの比較（2026年時点）：

  • Duo Enterpriseは$39/ユーザ/月で、脆弱性自動解決・根本原因分析・カスタムエージェントが含まれる
  • GitLab Ultimateを既に使っていればAI機能がサブスクリプションに含まれる（追加費用なし）
  • セキュリティスキャン・コンプライアンスがGitLab内で完結（外部ツール不要）
  • 自己管理型（Self-Managed）ではAI Gateway経由でコード・プロンプトを社内に閉じられる → 閉域優位
  • 弱点：C/C++特化の車載実績はCopilot比で少ない。Codebeamer公式連携の詳細は限定的

参照：GitLab Duo vs GitHub Copilot比較（OTTRA 2025）

車載開発での適合性： ★★★★☆（閉域・セキュリティ重視ならE/I構成が有力）

3.5 構成C：Amazon Kiro（将来候補）

Spec-Driven Development（仕様書→コード生成）を特徴とするAgentic IDE。VS Code fork。

  • C/C++対応確認済み。2025年11月GA
  • ALM要求仕様とのフィット感が高い可能性（仕様書からコードへの直接変換）
  • ただし車載・製造業の採用実績はほぼなし
  • SaaS接続前提（ファイアウォール/allowlist設定は可能）

参照：Amazon Kiro詳細解説（InfoQ）

車載開発での適合性： ★★★☆☆（PoC候補として注視。実績積み上げ待ち）

Part 5｜結論と推奨

5.1 推奨構成と理由

推奨：構成A（GHEC + Copilot Enterprise）から開始
  1. TISAX AL2認定取得済み → 自動車業界のセキュリティ要件に直接対応
  2. Mercedes-Benz・Woven by Toyota等のOEM実績が最も豊富
  3. Codebeamer ↔ GHEC連携はREST/Webhookで確立されたパターンが存在
  4. CodeQL（MISRA/AUTOSAR C++クエリ）がISO 26262ツール認定取得済み
  5. 導入難易度が最も低く、段階的拡張が容易

5.2 フェーズ展開案

Phase 1: GHEC + Copilot Enterprise（構成A）
  → SW部品単位のコード・テスト・PRの管理とAIコーディング支援を開始
  → Codebeamer ↔ GHEC Webhook連携でトレーサビリティを構築
  → Copilot Spacesで設計書ナレッジを蓄積

Phase 2: Claude Code追加（構成B）
  → 大規模リファクタリング・複数リポジトリ横断分析が必要になった段階で追加
  → MCP接続でCopilot（GHEC）とALM（Codebeamer）を論理的に統合

Phase 3: PoC（構成C / E / H検討）
  → 閉域要件が強まった場合にGitLab Self-Managed + Duo（構成I）を評価
  → 仕様書駆動開発に注力したい場合にKiro（構成C）をPoC

5.3 選定フローチャート

外部通信禁止（完全閉域）？
  └ YES → 構成H（GHES + Continue）または構成I（GitLab + Duo Self-Hosted）
  └ NO
      ├ GitLab既存ユーザ → 構成E（GitLab Duo）も評価対象
      ├ Azure既存ユーザ → 構成F1/F2（Azure DevOps + Copilot）も選択肢
      ├ 仕様書駆動開発を重視 → 構成C（Kiro）のPoC実施
      └ OEM実績・TISAX・導入容易性を重視
            ├ コードベース横断推論も必要 → 構成B（Copilot + Claude Code）
            └ まず標準的に始める → 構成A（Copilot）★最推奨
